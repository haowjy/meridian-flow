# Work Item Domain

Types and interfaces for work items — the grouping primitive that ties threads to shared artifact folders. Import: `meridian/internal/domain/workitem`. Deep dive: `.meridian/fs/backend/work-items/`.

## Key Concepts

- **Two statuses only**: `active` and `done`. Deletion is orthogonal — `deleted_at` timestamp, not a status value.
- **Ephemeral work items**: Auto-created when a thread needs a work item. Per-project cap of 100 with reuse on overflow.
- **Slug-addressed**: REST API uses project-scoped slugs, not UUIDs. Service generates slugs with collision retry.
- **CAS status transitions**: `UpdateStatus` uses compare-and-set — wrong current status returns `ErrConflict`.
- **Streaming guard**: `Complete` rejects if any thread has an in-flight streaming turn (`HasStreamingThreads`).

## Conventions

- Authorization lives in the Service layer — every mutating method takes `userID` and verifies project membership.
- `EnsureThreadWorkItem` is idempotent — returns existing work item if thread already has one.
- `Metadata` is full-replace on update, not deep-merge.
